Part 1: Why FastAPI for Machine Learning?
FastAPI is Python's leading framework for building APIs, and has become the go-to tool for deploying machine learning models. 9 out of 10 companies use FastAPI for their ML model APIs. This guide explains why — and how to use it confidently.
1.1  The ML Lifecycle — Where FastAPI Fits
A real-world ML product passes through four stages. FastAPI owns Stage 3: Deployment.

StageActivityTool1  Model DevelopmentBuilding ML / DL / GenAI modelsPython, Jupyter Notebook2  Model ValidationTesting for accuracy with metricsEvaluation metrics, test sets3  DeploymentBringing the model to customersFastAPI4  Frontend IntegrationConnecting web or mobile appAPI endpoints
?? TIP  A model that cannot reach users has zero business value — no matter how accurate it is. FastAPI bridges that gap.1.2  Why FastAPI Stands Out
• Blazing fast: Performance comparable to Node.js and Go.
• Auto-generated docs: Swagger UI and ReDoc pages built automatically from your code.
• Type safety: Built on Python type hints and Pydantic — data validation is built in.
• Async support: Native async/await for handling many concurrent requests.
• Easy to learn: A working API in under 10 lines of code.


Part 2: API Fundamentals
2.1  What Is an API?
API (Application Programming Interface) — a mechanism that enables two software components to communicate using a defined set of rules, protocols, and data formats.
• Connector: The bridge between two pieces of software.
• Protocol: Defines rules so both sides understand each other.
• Contract: Guarantees a predictable format for every request and response.

?? TIP  Think of an API like a power socket. You don't need to know how electricity is generated — you just plug in and get power. The API hides backend complexity and gives you a clean interface.2.2  The Request-Response Cycle
Every API interaction follows this flow (using an e-commerce search as the example):
1. User (Frontend): Types 'AI Agents' in the search bar and presses Enter.
2. HTTP Request: The browser sends a request to the API endpoint.
3. API Layer: Validates the request and forwards it to the backend.
4. Backend: Runs the search algorithm, queries the database, returns raw results.
5. API Layer: Formats the data as JSON and sends the HTTP response.
6. Frontend: Receives the response and displays results to the user.

Key Technical Elements
• Protocols — HTTP (standard web) or HTTPS (encrypted). Every API call travels over one of these.
• Data Format — JSON (JavaScript Object Notation) is most common. Human-readable, lightweight, universal.
• Endpoints — Specific URLs where the API lives, e.g. /search, /users, /orders.
2.3  The Restaurant Analogy

Restaurant RoleSoftware EquivalentFunctionCustomerFrontend (Web / Mobile)Where the user sits and makes requestsWaiterAPICarries requests and returns responses — uses JSON formatKitchen / ChefBackendWhere business logic and data processing happenFood OrderHTTP RequestWhat the client asks forPrepared DishHTTP Response (JSON)What the backend returns
?? NOTE  The customer never enters the kitchen and the kitchen never comes to the table. The API enforces that clean separation — this is what makes modern software scalable.

Part 3: Monolithic vs API-Based Architecture
3.1  The Monolithic Era
Before APIs, companies built monolithic applications — a single codebase containing frontend, backend, and database logic all tightly coupled together.
IRCTC Case Study: Building Without APIs
Imagine a railway booking website showing trains between two stations. A monolithic system has:

LayerDescriptionDatabaseAll train info, station data, schedules — one central repositoryBackendPython/Java; contains fetchTrains(station1, station2, date) functionFrontendWeb form with From, To, Date fields — directly calls backend functions
Key Characteristics of Monolithic Architecture
• Single Directory: All code in one folder (src/frontend, src/backend, etc.).
• Tightly Coupled: Frontend and backend are part of one application; cannot run independently.
• Direct Communication: Frontend directly calls backend functions — no separation.
3.2  Why Monolithic Architecture Failed
Problem 1: Third-Party Integration
Travel companies (MakeMyTrip, Yatra, Ixigo) approached IRCTC: 'We need train data for our users — we will pay per request.'

Option TriedProblemVerdictGive direct database accessSecurity risk — third parties could corrupt or delete dataNot feasibleExpose backend functionBackend is not independent; tight coupling prevents external accessNot possible
? WARNING  In monolithic architecture, you cannot share data or functionality with external parties — everything is tightly coupled in a single application.
Problem 2: Multi-Platform Expansion
Companies needed a website, Android app, and iPhone app — all showing the same data. Each platform required its own language and codebase:
• Android: Java
• iOS: Swift
• Web: HTML / CSS / JavaScript

ProblemImpact3× SynchronisationA comment on the website must appear on Android — constant manual syncing3× ResourcesThree development teams, three maintenance efforts, three infrastructure budgets3× Consistency bugsDifferent business logic per platform; feature parity breaks constantly3.3  The API Solution
Extract the backend into an independent application, place an API layer in front of it, and any frontend — web, Android, iOS — can communicate with the same backend:

Architecture ElementDescriptionSingle BackendOne database, one business logic layer, one source of truthAPI LayerA set of HTTP endpoints any client can call with the right credentialsMultiple FrontendsWebsite, Android, iOS — each hits the same API, gets the same data
• Single Source of Truth: Update once — reflects everywhere, instantly.
• Simplified Development: One backend team instead of three.
• Monetisation: API validates requests and can charge per call (MakeMyTrip pays IRCTC per search).
• Security: Backend is never directly exposed; API acts as a controlled, authenticated gateway.

Part 6: Path Parameters
6.1  What Are Path Parameters?
Path parameters are dynamic segments of a URL path used to identify a specific resource. Instead of creating a separate endpoint for each record, one route pattern handles all of them.

ApproachURL ExampleBehaviourWithout path parameter (Static)/patientsReturns ALL patients — cannot target oneWith path parameter (Dynamic)/patient/{patient_id}Returns only the patient whose ID matches
Basic Path Parameter Example
from fastapi import FastAPI

app = FastAPI()

@app.get('/patient/{patient_id}')   # {patient_id} is the path parameter
def view_patient(patient_id: str):  # FastAPI extracts it automatically
    data = load_data()
    if patient_id in data:
        return data[patient_id]
    return {'error': 'Patient not found'}  # <- we will improve this shortly6.2  Path Parameter Use Cases

OperationHTTP MethodEndpoint ExamplePurposeRetrieveGET/patient/{patient_id}Fetch a specific patientUpdatePUT/PATCH/edit/{patient_id}Modify a specific patient's recordDeleteDELETE/delete/{patient_id}Remove a specific patient6.3  The Path() Function — Validation and Docs
The Path() function replaces the plain type annotation and does three jobs: adds descriptions to auto-docs, validates incoming values, and rejects bad input before your function runs.
Path() Options Reference

OptionDescriptionExampledescriptionShown in Swagger UI docsdescription='ID of the patient'exampleSingle example in docsexample='P001'examplesMultiple examples in docsexamples=['P001', 'P002']geGreater than or equal toge=0  (no negatives allowed)gtStrictly greater thangt=0  (must be >= 1)leLess than or equal tole=1000ltStrictly less thanlt=1000min_lengthMinimum string lengthmin_length=3max_lengthMaximum string lengthmax_length=20regexRegular expression patternregex='^P[0-9]{3}$'
Example 1 — String Parameter with Description
from fastapi import FastAPI, Path, HTTPException

app = FastAPI()

@app.get('/patient/{patient_id}')
def view_patient(
    patient_id: str = Path(
        ...,                           # '...' means REQUIRED (no default)
        description='Patient ID — format: P followed by 3 digits',
        example='P001'
    )
):
    data = load_data()
    if patient_id in data:
        return data[patient_id]
    raise HTTPException(status_code=404, detail='Patient not found')
Example 2 — Integer with Numeric Range Validation
@app.get('/items/{item_id}')
def get_item(
    item_id: int = Path(
        ...,
        title='Item ID',
        description='The ID of the item to retrieve',
        ge=1,      # Must be >= 1  (rejects 0 and negatives)
        le=1000,   # Must be <= 1000
        example=42
    )
):
    return {'item_id': item_id}
Example 3 — String with Length and Regex Validation
@app.get('/user/{username}')
def get_user(
    username: str = Path(
        ...,
        min_length=3,                    # At least 3 characters
        max_length=20,                   # At most 20 characters
        regex='^[a-zA-Z0-9_]+$'         # Alphanumeric and underscore only
    )
):
    return {'username': username}
?? TIP  Use regex validation for IDs with a fixed format. regex='^P[0-9]{3}$' enforces that patient IDs must match P followed by exactly three digits: P001, P002 ... P999.

Part 7: HTTPException — Proper Error Handling
7.1  The Problem: Returning Errors as 200 OK
A common beginner mistake is returning an error message inside a plain dictionary. FastAPI sends it with a 200 OK status — which is misleading and breaks API clients.
# BAD PRACTICE — returns 200 OK even when patient does not exist!
@app.get('/patient/{patient_id}')
def view_patient(patient_id: str):
    data = load_data()
    if patient_id in data:
        return data[patient_id]
    return {'error': 'Patient not found'}  # Status is STILL 200
? WARNING  The client receives HTTP 200 (Success), but the body says 'Patient not found'. Client code checks the status code, thinks everything worked, then crashes or shows wrong data.7.2  The Correct Approach: raise HTTPException
from fastapi import FastAPI, HTTPException

app = FastAPI()

# GOOD PRACTICE — returns correct 404 status with a clear message
@app.get('/patient/{patient_id}')
def view_patient(patient_id: str):
    data = load_data()
    if patient_id in data:
        return data[patient_id]
    raise HTTPException(
        status_code=404,
        detail='Patient not found'
    )
ScenarioWithout HTTPExceptionWith HTTPExceptionPatient P001 exists200 OK + patient data200 OK + patient dataPatient P999 missing200 OK + {error: 'not found'}404 Not Found + {detail: 'Patient not found'}Client behaviour on errorMay silently fail or show wrong dataClient correctly handles the error case7.3  Common HTTPException Patterns
from fastapi import FastAPI, HTTPException, status

# 404 — Resource not found
raise HTTPException(status_code=404, detail='Patient not found')

# 400 — Bad / malformed request
raise HTTPException(status_code=400, detail='Invalid date format')

# 401 — Authentication required
raise HTTPException(status_code=401, detail='Authentication required')

# 403 — Authenticated but no permission
raise HTTPException(status_code=403, detail='You do not have permission')

# Using the status module for readability (recommended)
raise HTTPException(
    status_code=status.HTTP_404_NOT_FOUND,
    detail='Resource not found'
)
?? TIP  Always use raise HTTPException instead of returning error dictionaries. It makes your API predictable, self-documenting, and compatible with all API clients and libraries.

Part 8: Query Parameters
8.1  What Are Query Parameters?
Query parameters are optional key-value pairs appended to the end of a URL. They pass additional data to the server without altering the endpoint path — the standard mechanism for filtering, sorting, searching, and pagination.
URL Structure
http://localhost:8000/sort?sort_by=height&order=desc
                          ^              ^     ^
                       endpoint   key1=value1  key2=value2

?   separates the path from the query string
&   separates multiple key-value pairs
Use CaseExample Query StringWhat It DoesFiltering?city=Delhi&gender=maleShow only patients from Delhi who are maleSorting?sort_by=bmi&order=descSort patients by BMI in descending orderSearching?search=AnanyaSearch for patients named AnanyaPagination?page=2&limit=10Show 10 items on page 28.2  Implementing Query Parameters
Basic Sorting Endpoint
from fastapi import FastAPI, Query, HTTPException

app = FastAPI()

@app.get('/sort')
def sort_patients(
    sort_by: str = Query(
        ...,   # Required — no default value
        description='Sort on the basis of height, weight or bmi'
    ),
    order: str = Query(
        'asc',  # Optional — default is ascending
        description='Sort order: asc or desc'
    )
):
    data = load_data()
    valid_fields = ['height', 'weight', 'bmi']
    if sort_by not in valid_fields:
        raise HTTPException(status_code=400, detail=f'Invalid sort_by. Choose from: {valid_fields}')
    reverse = (order == 'desc')
    return sorted(data.values(), key=lambda x: x.get(sort_by, 0), reverse=reverse)
Example calls:
• GET /sort?sort_by=height              Sort by height ascending (default)
• GET /sort?sort_by=bmi&order=desc  Sort by BMI descending
• GET /sort?sort_by=name              400 Bad Request — invalid field

Filtering Endpoint
@app.get('/filter')
def filter_patients(
    city:    str = Query(None, description='Filter by city'),
    gender:  str = Query(None, description='male or female'),
    min_age: int = Query(None, ge=0,   description='Minimum age'),
    max_age: int = Query(None, le=120, description='Maximum age'),
):
    data = list(load_data().values())
    if city:                data = [p for p in data if p.get('city')   == city]
    if gender:              data = [p for p in data if p.get('gender') == gender]
    if min_age is not None: data = [p for p in data if p.get('age', 0) >= min_age]
    if max_age is not None: data = [p for p in data if p.get('age', 0) <= max_age]
    return data8.3  Path vs Query Parameters — When to Use Which

AspectPath ParameterQuery ParameterPosition/patient/{patient_id}/patients?city=DelhiRequired?Yes — part of the routeNo — can have a default (including None)Best forIdentifying a specific resourceFiltering, sorting, searching, paginationHTTP MethodsGET, PUT, DELETE on one itemGET on a collection
?? TIP  Targeting a SPECIFIC resource? Use a path parameter. Filtering or sorting a COLLECTION? Use query parameters.
